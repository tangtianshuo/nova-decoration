# 整体测试用例清单与 Playwright 规划

## 1. 目标与策略

- 目标：覆盖本项目“管理端 + 访客端 + 短链兜底 + 平台租户管理”的核心业务链路。
- 策略：先沉淀可执行的测试用例清单，再按优先级逐步自动化到 Playwright。
- 原则：主链路优先、风险优先、可回归优先。

---

## 2. 测试范围

- 管理端：登录、权限、公司资料、素材管理、展示页管理、分享二维码。
- 访客端：展示页访问、短链跳转、兜底页。
- 平台端：超级管理员租户管理与账号管理。
- API 与数据：关键接口契约、状态流转、多租户隔离。
- 非功能：安全、性能、稳定性、可观测性。

---

## 3. 测试环境与账号

- 前端：`pnpm dev --mode e2e`
- API：`pnpm dev:api`
- 数据：`pnpm run db:init && pnpm run db:seed`
- 租户管理员（seed）：
  - `admin@nova.local / 12345678`
- 平台管理员（seed）：
  - `superadmin@nova.local / 12345678`

## 6. Playwright 自动化落地（后续实施）

### 6.1 自动化分期

- P0（立即）：`AUTH-001/002/005/006`、`DASH-001`、`REDIRECT-001/002/003/004`、`FALLBACK-001/002/003`
- P1（第二阶段）：页面管理主流程、分享二维码主流程、公开展示访问、关键导航
- P2（第三阶段）：平台租户管理、上传签名与对象存储链路、越权与安全负向、事件统计

### 6.2 建议目录结构

```text
e2e/
  auth/
    auth-login.spec.ts
    auth-guard.spec.ts
  dashboard/
    dashboard-smoke.spec.ts
  pages/
    page-crud.spec.ts
    page-publish-offline.spec.ts
  share-links/
    share-link-create.spec.ts
    short-redirect.spec.ts
  fallback/
    fallback-reasons.spec.ts
  platform/
    platform-tenants.spec.ts
  fixtures/
    users.ts
    pages.ts
  utils/
    auth.ts
    api.ts
```

### 6.3 编写规范（建议）

- 每条自动化用例绑定一个或多个 `用例ID`，在测试标题中体现。
- 使用稳定定位器：优先 `getByRole`、`getByPlaceholder`、`getByTestId`。
- 对关键链路加截图/trace/video，仅在失败时保留。
- 对环境有依赖的用例分组执行：`mock-only`、`api-required`、`platform-admin`。

---

## 7. 回归准入标准

- P0 用例通过率 100%，P1 用例通过率 >= 95%。
- 不允许存在阻断级缺陷（登录失败、发布不可用、短链错误跳转）。
- 新功能上线必须至少补充 1 条对应自动化用例（或明确豁免理由）。
